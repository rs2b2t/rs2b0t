# Medium Clue Solver — design

**Date:** 2026-07-19 · **Status:** approved direction, spec for review
**Goal:** Extend the shipped easy-clue solver to also solve **medium** clue scrolls, in place (same obj-id-keyed DB, bank-first prep, executor, yield-on-event machinery).
**Scope (user-confirmed):** ALL medium mechanics — maps, coordinate/sextant digs, riddles (including kill-for-key combat), anagrams/speak-to, and challenge scrolls — EXCEPT automating the Observatory Quest + sextant/watch/chart *acquisition* (those items are assumed already obtained and kept in the bank). Risky combat targets are attempted uniformly with held gear (no special-casing) and abandoned honestly if unwinnable/unreachable.

## 1. Background — the easy solver (reused as-is)

- `src/bot/clues/data/cluedb.ts` (GENERATED by `tools/clues/gen-cluedb.ts` from the content pack): `CLUE_DB: Record<objId, ClueRow>` + `CASKET_IDS`. Each clue scroll is one obj id (all display "Clue scroll"; the id is the only client-readable discriminator; params aren't client-readable, so the table is load-bearing). Trails advance by the held scroll id swapping.
- `ClueRow` (`src/bot/clues/types.ts`): `{ obj, id, type: 'search'|'dig'|'talk', coord?, casketObj?, casketId?, npc? }`. `search`=walk to `coord` + search the loc; `dig`=walk to `coord` + dig → `casketObj`; `talk`=talk to `npc`.
- `SolveClue` (`src/bot/clues/SolveClue.ts`, a `Task`): bank-first ONCE per solve at the nearest bank — deposit all except the keep-set (clue/casket, food, spade, coins), ensure a spade + toll coins + food — then `ClueExecutor.solveHeldClue`. Outcomes `yield` (random event → hand back to loop), `abandon` (leave the clue in the pack, stop re-firing this session), or done.
- `ClueExecutor` (`src/bot/clues/ClueExecutor.ts`): identifies the held clue via the DB, walks to the target (`Traversal.walkResilient`), runs the search/dig/talk, opens reward caskets to advance, yields on `EventSignal.pending()`.
- Generation: `gen-cluedb.ts` reads `~/code/rs2b2t-content/scripts/minigames/game_trail/configs/trail_easy.{enum,obj}` for the clue list + `trail_coord` dig/search tiles, `.rs2` scripts (`parseTalkMappings`) for talk targets, `obj.pack`/`.npc` for ids/display names. Drift-gated: `bun tools/clues/gen-cluedb.ts --check`.
- Audit: `tools/clues/audit-clues.ts` — offline, asserts every clue resolves to a reachable, well-formed step (currently 66/66). Live smoke: `tools/cluesolve-test.ts`.

**Everything above is reused unchanged in shape.** Medium is new content-derived DATA + a few new executor branches.

## 3. Design

### 3.1 Data — extend `gen-cluedb` to `trail_medium`
Add a medium pass parallel to the easy one: parse `trail_medium.{enum,obj}` for the 56 clues + their `trail_coord`; reuse `parseTalkMappings` for anagram/speak-to NPCs; parse `trail_checkmediumdrop` for the kill-for-key NPC→key map; fold in a small curated **challenge Q→A** table (the maths answers — few, stable, not cleanly parseable). Emit medium rows into the same `CLUE_DB` (keyed by obj id — easy and medium ids are disjoint, so one table serves both). Regenerate `cluedb.ts`; `--check` drift gate covers medium too.

### 3.2 Types — small additive fields on `ClueRow`
- `dig` gains `needsSextant?: boolean` — true for the 20 sextant clues (require sextant+watch+chart held).
- `search` gains `keyFrom?: { npc: string; keyObj: string; keyId: number }` — the kill-for-key riddles: kill `npc`, loot `keyObj`, then search.
- A step may carry `challenge?: true` (or the executor detects the challenge dialogue generically) → answer from the curated table.
- `talk` unchanged (anagram/speak-to).
`ClueType` stays `search|dig|talk`; the new behavior rides on the additive fields, so easy rows are untouched and the audit/executor branch only when a field is present.

### 3.3 Executor — new branches (`ClueExecutor`)
- **Coordinate dig** (`dig` + `needsSextant`): if sextant/watch/chart aren't all held → `abandon` with a clear reason ("no sextant/watch/chart — coordinate clue skipped"); else dig at `coord` exactly like an easy map dig.
- **Kill-for-key** (`search` + `keyFrom`): if the key isn't already held, walk to the NPC, attack it with held gear, wait for the kill, loot the dropped key (ground item), then walk to `coord` and search the container. Idempotent/re-entrant: re-identify checks "key held?" so a mid-fight yield resumes cleanly. Bounded — if the NPC can't be reached/killed within a budget, `abandon`.
- **Challenge scroll**: when a talk/search step surfaces a challenge dialogue (a maths question), answer with the looked-up number and continue driving; if unrecognised, `abandon`.
- **Anagram/speak-to** (`talk`): existing path unchanged.

### 3.4 Bank-first keep-set (`SolveClue.bankFirst`)
Add to the keep-set: **sextant, watch, chart** (coordinate clues) and a **weapon + food** allowance for the combat riddles (food is already kept; add the configured weapon name). The host's `foodName`/`spadeName` pattern extends with a `weaponName()`/coordinate-item names. Coordinate/combat items are best-effort: absent → the dependent clue abandons, the rest of the trail proceeds.

### 3.5 Testing
- **Offline audit** (`audit-clues.ts`): extend to assert every MEDIUM clue resolves to a well-formed step — dig/search tiles present + reachable (pack-gated pathfind), talk NPCs resolved, kill-for-key rows carry a known NPC + key id, sextant rows flagged, challenge answers present. Target: all easy + all medium clean.
- **`--check` drift gate** stays green (medium rows included).
- **Live smoke** (`cluesolve-test.ts` or a medium variant): drive a representative sample end-to-end against the local engine — one map dig, one coordinate dig (with sextant/watch/chart given), one kill-for-key riddle, one anagram talk, and a challenge scroll — each reaching its reward casket.

## 4. Out of scope
- Automating the **Observatory Quest** and the sextant/watch/chart acquisition chain (Murphy/Kojo/assistant). Coordinate clues assume the three items exist; absent → that clue abandons.
- Hard-tier clues (a later, separate spec).
- Reading the sextant interface to *derive* a coordinate (unneeded — the content stores the dig tile).

## 5. Risks & mitigations
- **Content-derivation gaps** (a medium clue whose `trail_coord`/talk target the generator can't resolve, like easy's `vague003`/`_sailor` special-cases): handle with the same curated `specials`/alias map in `gen-cluedb`; the audit catches any unresolved row before it ships.
- **Combat variance** (kill-for-key targets differ in difficulty; Black Heather is in the low wilderness): attempt uniformly with held gear, bounded, `abandon` on failure — no special risk-gating (per user). Wilderness exposure is the honest `abandon` path, not a crash.
- **Shared executor regression**: the new branches are gated on the additive fields, so easy-clue behavior is unchanged; the audit + a re-run of the easy live smoke guard it.
- **Challenge Q→A staleness**: curated table; the audit asserts each known challenge has an answer, and an unrecognised challenge abandons rather than mis-answers.
